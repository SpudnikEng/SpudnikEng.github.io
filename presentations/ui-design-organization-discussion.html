Why “high level” first?

Organization (information architecture, navigation, modes, what is primary vs secondary) is expensive to change late. Visual polish is comparatively cheap.

If we skip straight to controls…

  • Experts get screens that mirror the machine map, not the job
  • Operators hunt for the one number that matters right now
  • Programmers inherit ambiguous state boundaries

If we align on structure first…

  • Sales can train consistent “paths” through the UI
  • Operators build muscle memory across machines
  • Reuse (shell, patterns) becomes realistic across product lines
Discuss
  • Where have we paid the price for late structural changes?
  • What is one “organization” win we are already proud of?

Shared vocabulary

  • Information architecture (IA): how content is grouped, named, and reached (navigation), not individual widget styling.
  • Primary task surface: the screen (or region) tuned to what the operator is doing most of the time in the field.
  • Progressive disclosure: show essentials first; reveal depth (diagnostics, tuning) on demand.
  • Affordance: what the control “looks like it does” — especially important under vibration and gloves.
  • Mode: a machine or UI state that changes which actions are legal and what is shown (e.g. transport vs field, auto vs manual).
  • Alarm vs warning vs status: urgency and required response differ — the UI should not flatten them into one color.
Discuss
  • Which term is most often misunderstood between departments?
  • Do we need a one-page glossary for customers?

Three lenses → one UI contract

Everyone touches the same HMI for different reasons. Design decisions should be defensible from all three angles.

Operator
safety, glanceability, flow under load
+
Sales / setup
language, training, setup order
+
Programmer
states, boundaries, reuse

→ One documented UI contract (IA + modes + naming)

Operator

“Can I see the problem and the next safe action without reading a manual?”

Sales / setup

“Can I teach this in one cab ride using the same words the customer uses?”

Programmer

“Can I implement this without guessing hidden states or duplicate sources of truth?”

Discuss
  • When have two lenses agreed but the third broke the design?
  • Who owns keeping the “UI contract” document current?

Principle: task-first layout

Structure around the job in the cab (dig / pick / transfer / unload for a harvester; populate / depth / sections / material for a planter), not a flat list of every subsystem.

Harvester angle

High rate of change; attention is split between field, machine noise, and display. Primary surface should answer: “Is the machine healthy and producing?” and “What do I adjust next?”

Planter angle

More setup and verification upfront. Primary surface in the field often answers: “Is each row doing what we agreed in setup?”

Operator

Minimize navigation during the highest-stress minutes.

Sales / setup

Teach one “happy path” that matches the task story.

Programmer

Map screens to explicit machine states and transitions.

Discuss
  • What is the true “primary task” for each machine in the first hour of operation?
  • What secondary features keep creeping onto the main screen?

Same principle, two machines

Principle: one primary metric (or small cluster) plus a context strip (status, mode, next risk).

Harvester example

Primary might emphasize throughput / loss / blockage risk — signals that change quickly and drive immediate response.

Planter example

Primary might emphasize population / skip indicators / depth stability — slower drift, row-by-row meaning.

The shell (how you navigate, acknowledge alarms, see mode) can be shared; the hero region should respect different operator attention patterns.

Discuss
  • What is the real primary metric for your customers on each machine?
  • Would forcing the same primary metric across both machines hurt usability?

Modes & safe boundaries

When the machine or UI mode changes, the display should make that obvious: different actions are legal, different risks apply (transport vs field, manual valve control vs interlocks, etc.).

Operator

Never surprise me with available actions — mode changes should be loud and reversible where safe.

Sales / setup

Training scripts should name modes the same way the UI does.

Programmer

Illegal actions are disabled and explained when possible; state charts stay traceable.

Discuss
  • List the modes that must never be confused on each machine.
  • Where have we hidden mode in a small icon when it should dominate?

Progressive disclosure

Run surfaces stay shallow. Diagnostics and expert tuning earn their depth — but emergency and safety-related paths must stay shallow by design.

Harvester

During a plug or sensor fault, depth is acceptable after the operator sees location + severity + suggested first step.

Planter

Calibration wizards can be deep; mid-shift adjustments for one row should not require backing out of five menus.

Discuss
  • What is one thing we buried too deep on a current product?
  • What expert page is used often enough that it should be promoted?

Alarms & faults

Operators rarely want an encyclopedia first — they want grouping, acknowledgment that sticks, and “fix next” guidance. Sales cares that training matches alarm text. Programmers need stable IDs and severity rules.

Operator

Can I tell new vs standing fault? Is silence informative or dangerous?

Sales / setup

Do customer-facing words match what appears on screen?

Programmer

Are alarm codes, help text, and PLC logic a single traceable chain?

Discuss
  • What do operators actually do first when the list has ten items?
  • How do we avoid “alarm fatigue” without hiding real risk?

Setup vs runtime information architecture

Setup / commissioning

Wizards, checklists, and “done” states. Planter-heavy. Programmers need explicit entry/exit and persistence rules.

Runtime

Dashboards, quick tweaks, alarm handling. Harvester often stresses continuous adjustment. Keep the path from run → safe stop → diagnosis obvious.

Discuss
  • Do we use the same navigation entry for “setup” on both machines?
  • Where does runtime accidentally expose dangerous setup controls?

Recap: principles checklist

  • Task-first primary surface (job story, not subsystem map)
  • Harvester vs planter: same shell ideas, different hero metrics
  • Modes visible; illegal actions explained where feasible
  • Progressive disclosure without burying emergencies
  • Alarms: grouping, ack behavior, operator-first narrative
  • Setup vs runtime paths are intentionally different depths
  • Terminology bridge is maintained and owned
Discuss
  • Strike any principle that does not fit our current reality — why?
  • Add one missing principle the team believes in.
